# Architecture Scope: Annotation Markdown Exchange

**Дата:** 22.04.2026  
**Статус:** Proposed  
**Issue:** [#5](https://github.com/bon2362/reader/issues/5)

## 1. Цель

Добавить переносимый пользовательский сценарий для:

- массового экспорта аннотаций по всем книгам библиотеки;
- обратного импорта ранее экспортированных аннотаций;
- безопасного повторного импорта без размножения дублей.

В отличие от резервной копии `SQLite`, это должен быть человекочитаемый и контролируемый формат обмена.

## 2. Главное решение

### Оставляем только Markdown

Формат `.txt` исключается из scope как избыточный.

Причина:

- `Markdown` уже покрывает читаемость человеком;
- `Markdown` позволяет встроить служебные поля для машинного импорта;
- один формат проще тестировать, версионировать и поддерживать.

Итоговое правило:

- экспортируем только `.md`;
- импортируем только `.md`;
- `.md` является каноническим exchange-форматом аннотаций.

## 3. Принцип формата

Экспортируемый `.md` должен одновременно быть:

- удобным для чтения человеком;
- достаточно строгим для надёжного парсинга программой.

Для этого документ делится на две части:

- видимая часть: заголовки, цитаты, текст заметок, секции по типам аннотаций;
- служебная часть: машинные метаданные в YAML front matter и HTML-комментариях.

### Почему не свободный Markdown

Свободный пользовательский Markdown нельзя надёжно импортировать обратно:

- невозможно стабильно определить книгу;
- невозможно однозначно восстановить anchor;
- невозможно отличить новую аннотацию от уже импортированной;
- невозможно безопасно обновлять существующие записи.

Поэтому импорт поддерживается только для файлов, которые соответствуют формату `reader-annotations/v1`.

## 4. Границы MVP

### In scope

- Один `.md`-файл на книгу.
- Массовый экспорт всех книг библиотеки.
- Импорт одного или нескольких ранее экспортированных `.md`-файлов.
- Типы аннотаций:
  - `highlight`
  - `text_note`
  - `sticky_note`
- Preview перед импортом.
- Повторный импорт без дублей.

### Out of scope

- Импорт произвольного пользовательского Markdown.
- Экспорт/импорт `TXT`.
- Merge нескольких разных экспортов одной и той же книги в рамках сложной ручной reconciliation UI.
- Восстановление книги по аннотациям, если самой книги нет в библиотеке.

## 5. Формат файла

## 5.1 Один файл = одна книга

Экспорт по библиотеке создаёт набор `.md`-файлов, по одному на книгу.

Рекомендованная структура:

```text
Annotations Export/
  Book Title 1.md
  Book Title 2.md
  Book Title 3.md
```

Не рекомендуется один общий файл на всю библиотеку:

- сложнее preview;
- сложнее частичный re-import;
- выше риск конфликтов и плохих merge diff;
- хуже пользовательский контроль.

## 5.2 Заголовок файла

В начале каждого файла размещается YAML front matter:

```md
---
format: reader-annotations/v1
exportedAt: 2026-04-22T12:00:00Z
book:
  id: 8A1F...
  title: Example Book
  author: Jane Doe
  format: pdf
  contentHash: 3f2f...
---
```

### Назначение полей

- `format` — версия exchange-контракта.
- `exportedAt` — время создания файла экспорта.
- `book.id` — локальный идентификатор книги на момент экспорта, informational only.
- `book.title` / `book.author` — удобочитаемые поля.
- `book.format` — `epub` или `pdf`.
- `book.contentHash` — основной ключ сопоставления книги при импорте.

### Правило сопоставления книги

Во время импорта книга ищется прежде всего по `book.contentHash`.

`book.id` не используется как главный ключ импорта, потому что локальные id могут различаться между устройствами и установками.

## 5.3 Структура секций

Файл содержит три верхнеуровневые секции:

```md
# Annotations

## Highlights
...

## Text Notes
...

## Sticky Notes
...
```

Если аннотаций конкретного типа нет, секция может отсутствовать.

## 5.4 Формат отдельной аннотации

Каждая аннотация оформляется как видимый блок Markdown и скрытый HTML-комментарий с метаданными.

Пример `highlight`:

```md
### Highlight
<!--
id: exp-01HXYZ...
type: highlight
anchor:
  scheme: cfi
  value: /6/2[chapter1]!/4/2,/1:10,/1:24
createdAt: 2026-04-22T10:00:00Z
updatedAt: 2026-04-22T10:05:00Z
color: yellow
selectedText: Important quoted text
-->

> Important quoted text
```

Пример `text_note`:

```md
### Text Note
<!--
id: exp-01HXYZ...
type: text_note
anchor:
  scheme: cfi
  value: /6/2[chapter1]!/4/2,/1:10,/1:24
createdAt: 2026-04-22T10:06:00Z
updatedAt: 2026-04-22T10:07:00Z
selectedText: Important quoted text
-->

**Selected text**

Important quoted text

**Note**

My comment here.
```

Пример `sticky_note`:

```md
### Sticky Note
<!--
id: exp-01HXYZ...
type: sticky_note
anchor:
  scheme: page
  value: 17
createdAt: 2026-04-22T10:08:00Z
updatedAt: 2026-04-22T10:08:00Z
pageLabel: Chapter 2 · Page 4
-->

**Location**

Chapter 2 · Page 4

**Note**

Remember this section.
```

## 6. Anchor contract

Импорт невозможен без устойчивого указания, куда аннотация привязана.

Поэтому каждая аннотация обязана содержать `anchor`.

### Для EPUB

- `scheme: cfi`
- `value`: строковый anchor текущего EPUB-потока

### Для PDF highlights / text notes

- `scheme: pdf-anchor`
- `value`: текущий сериализованный PDF anchor из приложения

### Для PDF sticky notes

- `scheme: page`
- `value`: page index или другой текущий page-based identifier

### Архитектурное правило

Экспорт не должен придумывать новый anchor-формат.

Он обязан использовать уже существующий внутренний anchor той же сущности. Exchange-слой сериализует его как строку и возвращает обратно без дополнительной семантической интерпретации.

Это снижает риск потери точности между экспортом и импортом.

## 7. Identity и дедупликация

Это ключевая часть импорта.

### 7.1 Exchange ID

Каждая экспортированная аннотация получает поле `id`.

Это не UI-label и не обязательно равен текущему локальному database id. Это постоянный внешний идентификатор exchange-записи.

Требование:

- один и тот же экспортируемый объект должен сохранять один и тот же `id` между экспортами;
- повторный импорт того же файла не должен создавать дубли.

### 7.2 Рекомендуемая реализация

Добавить в локальные модели аннотаций поле наподобие:

- `exchange_id TEXT NULL`

Правила:

- для новых локальных аннотаций `exchange_id` генерируется один раз и хранится в базе;
- при экспорте наружу уходит именно `exchange_id`;
- при импорте поиск совпадений сначала идёт по `exchange_id`.

### 7.3 Если `exchange_id` отсутствует

Для legacy данных допустим fallback:

- сгенерировать `exchange_id` при первом экспорте;
- сохранить его локально;
- начиная с этого момента использовать как постоянный внешний id.

## 8. Как работает импорт

Импорт идёт в четыре последовательных шага.

## 8.1 Parse

Система читает `.md` и преобразует его в промежуточную exchange-модель:

- данные книги;
- список highlights;
- список text notes;
- список sticky notes.

Если:

- отсутствует `format`;
- версия не поддерживается;
- отсутствуют обязательные поля;
- структура аннотации повреждена,

то файл помечается как invalid и не импортируется.

## 8.2 Match Book

Система ищет книгу в библиотеке по `book.contentHash`.

Если книга не найдена:

- импорт аннотаций для этого файла не выполняется;
- файл попадает в preview/report как skipped;
- пользователь видит причину: matching book not found.

Мы не создаём “аннотации в пустоту” и не создаём автоматически новую книгу только на основании `.md`.

## 8.3 Dedupe / Resolve Existing

Для каждой аннотации система ищет существующую локальную запись:

1. сначала по `exchange_id`;
2. если не найдено и это legacy-path, допускается ограниченный fallback по типу + книге + anchor.

Fallback нужен только как миграционный мост. Основной путь должен быть через `exchange_id`.

## 8.4 Upsert

Если запись не найдена:

- создаём новую локальную аннотацию.

Если запись найдена:

- обновляем существующую.

### Правило конфликта

Для MVP применяем простое правило:

- выигрывает запись с более новым `updatedAt`.

То есть импорт не должен безусловно перетирать более новую локальную правку старым файлом.

## 9. Preview перед импортом

Импорт не должен сразу писать в БД.

Сначала нужен preview:

- сколько файлов распознано;
- сколько книг сопоставлено;
- сколько аннотаций будет создано;
- сколько обновится;
- сколько пропустится;
- сколько файлов невалидны.

Это уменьшает риск неожиданных изменений и делает импорт объяснимым пользователю.

## 10. Транзакционные границы

Импорт должен применяться транзакционно по книге.

Правило:

- одна книга = одна транзакция.

Если одна аннотация внутри конкретной книги ломает импорт этой книги:

- вся книга откатывается;
- другие книги могут импортироваться независимо.

Это лучше, чем одна глобальная транзакция на весь пакет:

- меньше blast radius;
- легче повторять импорт;
- проще объяснять ошибки.

## 11. Предлагаемые компоненты

### Exchange layer

- `AnnotationExchangeDocument`
- `AnnotationExchangeBook`
- `AnnotationExchangeItem`

### Export side

- `AnnotationExportService`
- `MarkdownAnnotationEncoder`

### Import side

- `MarkdownAnnotationDecoder`
- `AnnotationImportPreviewService`
- `AnnotationImportService`

### Repository boundary

Импорт не должен писать в `GRDB` напрямую из UI-слоя.

Он должен использовать существующие repository abstraction'ы и, при необходимости, расширить их upsert-операциями.

## 12. Правила совместимости

Формат обязательно versioned:

- `reader-annotations/v1`

Если позже структура расширится:

- добавляем `v2`;
- старый импортёр может либо поддерживать обе версии, либо явно отклонять неизвестную.

Нельзя делать “неявно плавающий” формат без версии.

## 13. Коротко для реализации

Если описать импорт совсем простыми словами, контракт такой:

1. Открыли `.md`.
2. Проверили, что это наш формат.
3. Поняли, для какой книги он создан.
4. Нашли эту книгу в библиотеке.
5. Для каждой аннотации проверили: она уже есть или нет.
6. Если нет — создали.
7. Если есть — обновили только если импортируемая версия новее.
8. Всё применили транзакцией на уровне одной книги.

## 14. Рекомендованный следующий шаг

Следующий логичный шаг после этого контракта:

- завести implementation story на schema support для `exchange_id`;
- затем отдельную story на export encoder;
- затем story на import preview + import apply.
