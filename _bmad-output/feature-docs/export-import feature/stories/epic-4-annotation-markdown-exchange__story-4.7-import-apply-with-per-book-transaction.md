# Story 4.7: Import Apply with Per-Book Transaction

**Epic:** 4 — Annotation Markdown Exchange  
**Status:** proposed  
**Created:** 2026-04-22

---

## Story

Как пользователь, я хочу применить импорт Markdown-аннотаций в библиотеку предсказуемо и без дублей, чтобы новые записи создавались, существующие корректно обновлялись, а ошибка в одной книге не ломала импорт остальных.

## Acceptance Criteria

- AC-1: Создан `AnnotationImportService`, который применяет результат импорта после preview
- AC-2: Для каждой книги импорт выполняется отдельной транзакцией
- AC-3: Совпадение существующей записи определяется прежде всего по `exchangeId`
- AC-4: Если запись не найдена, создаётся новая локальная аннотация
- AC-5: Если запись найдена, она обновляется только если импортируемый `updatedAt` новее локального
- AC-6: Повторный импорт того же файла не создаёт дублей
- AC-7: Ошибка импорта одной книги откатывает только эту книгу и не блокирует другие книги в пакете
- AC-8: Integration tests покрывают create, update, re-import и partial failure scenarios

## Dev Notes

### Это финальная story эпика

К этому моменту уже должны существовать:

- схема `exchangeId`;
- repository lookup support;
- exchange domain model;
- encoder;
- decoder;
- import preview.

Эта story соединяет всё это в рабочий apply flow.

### Главный принцип

Импорт должен быть idempotent.

То есть:

- один и тот же `.md`, импортированный повторно, не создаёт дубли;
- уже существующая запись не размножается;
- конфликт “старый файл против более новой локальной правки” решается предсказуемо.

### Match policy

Основной путь поиска существующей записи:

1. `bookId + exchangeId`

Не нужно в MVP делать сложные fallback heuristics по тексту, anchor и времени, если primary lookup не нашёл запись.

Если понадобится limited fallback для legacy-case, его лучше вынести отдельно и явно задокументировать, а не смешивать с основным путём.

### Conflict rule

Для MVP фиксируем простое правило:

- побеждает более новый `updatedAt`.

Это означает:

- если импортируемая запись новее локальной, обновляем;
- если локальная новее, импортируемую не применяем;
- если они эквивалентны, можно считать это no-op.

### Transaction boundary

Очень важно не делать один гигантский write на весь пакет.

Правильная граница:

- одна книга = одна транзакция.

Это даёт:

- меньший blast radius;
- понятные ошибки;
- возможность частичного успеха пакета.

### Где потребуется особая аккуратность

`TextNote` может быть логически связана с `Highlight` через `highlightId`, но в import/export контуре опираться на локальный `highlightId` между системами нельзя.

Для MVP pragmatic path:

- импортировать `TextNote` как самостоятельную запись по её собственному `exchangeId` и `anchor`;
- локальный `highlightId` восстанавливать только если такая связь уже доступна внутри текущего импортируемого набора и может быть установлена безопасно.

Если это усложняет apply слишком сильно, допустимо оставить `highlightId = nil` и не блокировать импорт заметки.

### Что не входит в эту story

- сложный reconciliation UI;
- ручное разрешение конфликтов;
- undo import;
- auto-import arbitrary Markdown.

### Test Guidance

Нужны полноценные integration tests на:

- create path для новой книги с несколькими типами аннотаций;
- re-import того же документа;
- local-newer wins;
- две книги в пакете, одна проходит, вторая откатывается.
