# Ретроспектива Epic 4: Annotation Markdown Exchange

**Дата:** 2026-04-23  
**Эпик:** Epic 4  
**Статус эпика:** done  
**Статус ретроспективы:** done

## 1. Epic Review

### Что было целью

Сделать переносимый и понятный контур обмена аннотациями через Markdown:

- экспорт одной книги в один `.md`;
- массовый экспорт по библиотеке;
- preview перед импортом;
- идемпотентный import без дублей;
- безопасное применение импорта по книгам;
- человекочитаемый формат, пригодный и для пользователя, и для машинной обработки.

### Что в итоге получилось

Epic достиг целевой функциональности и был дополнительно отполирован после основной реализации:

- реализован exchange-контур на базе `reader-annotations/v1`;
- добавлены `exchangeId` и lookup по `bookId + exchangeId`;
- реализованы encoder/decoder и независимая exchange-модель;
- добавлены preview и apply import с per-book transaction;
- доведён UI массового и точечного экспорта;
- исправлены реальные UX-ошибки, найденные уже после основной реализации:
  - сохранение `selectedText` у text notes;
  - корректный markdown-вывод контекстных секций;
  - корректная локация sticky note в UI и экспорте;
  - разделение экспорта текущей книги и всей библиотеки.

### Что сработало хорошо

- Разбиение на маленькие stories оказалось удачным. Фундамент (`exchangeId`, repository lookup, exchange-модель) действительно упростил следующие stories, а не создал лишний слой.
- TDD/targeted-test подход окупился. Самые рискованные зоны эпика были закрыты точечными тестами: migration, encoder/decoder, preview classification, apply semantics, export orchestration.
- Решение импортировать по `contentHash`, а аннотации матчить по `exchangeId`, оказалось устойчивым и хорошо объяснимым.
- Пер-book transaction для import apply дал правильную границу отказа: одна проблемная книга не ломает весь пакет.
- Дополнительная ручная проверка после завершения разработки выявила реальные продуктовые несоответствия, которые тесты изначально не ловили.

### Что было сложнее, чем ожидалось

- Реальная сложность оказалась не только в import/apply, а в “стыках” между моделью, UI и экспортным представлением.
- Формально story 4.5 была про library-wide export, но на практике пользователю понадобилось разделение на “экспорт текущей книги” и “экспорт всей библиотеки”. Это продуктовый нюанс, который не был до конца выражен в исходных acceptance criteria.
- Sticky notes показали системную проблему: внутренние технические координаты (`spineIndex`, `pageInChapter`) не всегда совпадают с пользовательским понятием страницы и главы.
- Xcode/project integration снова была частью сложности: новые файлы и миграции не всегда подхватывались целевым target автоматически, из-за чего часть решений пришлось встраивать в уже существующие файлы.

### Главные уроки

#### 1. Exchange-фичи нужно проектировать как продуктовый контур, а не только как сериализацию

Самая важная часть задачи была не в том, чтобы “сохранить данные в markdown”, а в том, чтобы пользователь мог доверять этому контуру:

- локация должна быть понятной;
- экспорт из UI должен соответствовать месту, где пользователь находится;
- повторный import должен быть предсказуемым;
- preview и apply должны совпадать по смыслу.

#### 2. Для аннотаций нельзя опираться только на локальные связи и внутренние идентификаторы

Связь `highlightId`, raw `spineIndex` и локальные page anchors подходят для внутреннего хранения, но плохо работают как межсистемный контракт. Для exchange-слоя нужны более устойчивые идентификаторы и более человеко-понятные labels.

#### 3. После завершения “основной” story-цепочки нужен обязательный polishing pass

Фича была функционально завершена, но именно ручная проверка после merge выявила наиболее заметные UX-проблемы. Это говорит о том, что для пользовательских фич import/export стоит заранее планировать короткий post-implementation hardening этап.

### Повторяющиеся паттерны и наблюдения

- Сильная сторона эпика: каждый следующий слой переиспользовал предыдущий без крупного рефакторинга.
- Повторяющийся источник риска: расхождение между внутренней технической моделью и тем, что видит пользователь в UI или markdown.
- Повторяющийся источник трения: Xcode target membership и project wiring для новых артефактов.
- Качество выросло заметно после реальных end-to-end сценариев, а не только unit/integration тестов.

### Технический долг и остаточные риски

- Import/export всё ещё завязан на локальный файловый UX и системные file panels; автоматизированного e2e-покрытия пользовательского flow пока нет.
- Формат `pageLabel` остаётся производным полем. Он полезен для UX, но его надо и дальше рассматривать как display-layer, а не как canonical anchor.
- Пост-epic улучшения были внесены уже после завершения основных stories. Это не критично, но в будущем лучше выделять такие доработки в явный hardening tail внутри эпика.

## 2. Next Epic Preparation

### Что стоит сохранить как рабочую практику

- Продолжать строить exchange- и sync-функции поверх отдельных domain-моделей, а не напрямую поверх UI/store слоёв.
- Оставить targeted test strategy как стандарт для рисковых историй с миграциями, сериализацией и reconciliation logic.
- Сразу планировать ручной checkpoint после завершения “основной” реализации и до окончательного закрытия эпика.

### Что нужно делать иначе в следующем эпике

- На этапе story decomposition отдельно выделять продуктовые сценарии “one item” vs “batch flow”.
- Для любых user-facing labels заранее фиксировать источник истины: технический идентификатор, глобальная страница, chapter label или вычисляемый display label.
- Если решение требует новых файлов в target, сразу проверять project wiring, а не только компиляцию логики.

### Action Items

- **Developer:** Для следующих exchange/sync-эпиков добавлять явный hardening checklist: manual UX pass, export/import smoke pass, consistency check между UI и serialized output.
- **Developer:** При появлении новых display-label полей вводить общий formatter/mapper раньше, а не после первых багов в UI.
- **Project Lead:** При формулировке следующих stories явно разделять batch UX и single-item UX, если оба режима важны пользователю.
- **Project Lead:** Добавлять в acceptance criteria хотя бы один пользовательский сценарий ручной проверки итогового артефакта, а не только внутреннего состояния системы.

### Решение по эпикe

Epic 4 можно считать завершённым не только по статусам stories, но и по фактической пользовательской проверке. Команда может переходить к следующему эпику без открытых блокеров по import/export контуру.
